# WhatsApp Web CRM Prototype

## English

WhatsApp Web CRM Prototype is a React, TypeScript, Supabase, and WhatsApp Web based CRM prototype for collecting WhatsApp leads, viewing message history, and managing a simple sales pipeline.

This repository is suitable for local development and GitHub-based continuation. It is not production-ready yet because it uses WhatsApp Web automation and demo-level Supabase Row Level Security policies that must be tightened before public deployment.

### System Requirements

- Node.js 18 or newer.
- npm 9 or newer.
- Supabase project with PostgreSQL enabled.
- Modern browser.
- A WhatsApp account for local bridge testing.

### Installation

```bash
npm install
cp .env.example .env
cd bridge
npm install
cp .env.example .env
```

Set the frontend variables in `.env`:

```ini
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=your-supabase-anon-key
```

Set the bridge variables in `bridge/.env`:

```ini
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_SERVICE_ROLE_KEY=your-supabase-service-role-key
```

### Database Setup

Apply the Supabase migration:

```bash
supabase db push
```

If Supabase CLI is not configured, copy the SQL from `supabase/migrations/20260623120653_create_crm_schema.sql` into the Supabase SQL editor and run it manually.

### Local Run

Run the frontend:

```bash
npm run dev
```

Run the WhatsApp bridge in a second terminal:

```bash
cd bridge
npm start
```

Open the Vite URL shown in the terminal, usually `http://localhost:5173`.

### Usage

1. Open the CRM dashboard.
2. Start the bridge with `cd bridge && npm start`.
3. Open the WhatsApp connection view.
4. Scan the QR code with WhatsApp on a phone.
5. Incoming WhatsApp messages are saved as leads and messages in Supabase.
6. Use the leads and pipeline views to inspect contacts and sales activity.

### Frontend Documentation

The frontend is a Vite React application written in TypeScript. It uses Tailwind CSS for styling, Lucide React icons, and `@supabase/supabase-js` for realtime database access.

Main screens:

- Dashboard with CRM statistics, recent leads, and recent messages.
- Leads list with search and lead navigation.
- Pipeline view for sales stages.
- WhatsApp connection view with QR/status display.
- Lead detail view for contact and message history.

### Backend and Bridge Documentation

There is no traditional server-side API in the frontend app. Data access currently happens through Supabase client calls.

The local bridge in `bridge/index.js` uses `whatsapp-web.js`, Puppeteer, and a Supabase service role key to:

- create or update the WhatsApp session row;
- store QR data for the frontend;
- create leads from incoming WhatsApp messages;
- store incoming and outgoing message records;
- update lead activity logs.

### API Documentation

The project uses Supabase tables directly:

- `leads`
- `messages`
- `pipelines`
- `pipeline_stages`
- `lead_activities`
- `whatsapp_sessions`

The Supabase Edge Function at `supabase/functions/whatsapp/index.ts` is present as a project hook, but the current working flow is the local Node bridge.

### Data Storage

PostgreSQL in Supabase stores leads, messages, pipeline data, lead activities, and WhatsApp session state. WhatsApp Web local authentication/cache files are created under `bridge/.wwebjs_auth/` and `bridge/.wwebjs_cache/`; these folders are ignored by Git and must not be committed.

### Forms and Email

The project does not include email sending, SMTP configuration, or contact form email delivery. If notifications are added later, store SMTP/API credentials in protected environment variables only.

### Security Notes

- Do not commit `.env`, Supabase service role keys, WhatsApp session folders, logs, database dumps, or real customer data.
- The Supabase service role key belongs only in `bridge/.env` or protected server configuration.
- The current migration enables broad anon/authenticated policies for prototype testing. Before production, replace them with tenant-aware RLS and authenticated access rules.
- WhatsApp Web automation is not the official WhatsApp Business API. For production, prefer the official WhatsApp Business Platform.
- Treat message contents and phone numbers as personal data.
- Use HTTPS for deployed frontend and bridge endpoints.

### Deployment Notes

Frontend deployment can use Vite static hosting:

```bash
npm run build
```

Upload `dist/` to static hosting and configure `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` during build.

The bridge should run on a protected server or process manager and must receive `SUPABASE_SERVICE_ROLE_KEY` from secure environment configuration. Do not expose the bridge service role key to the browser.
